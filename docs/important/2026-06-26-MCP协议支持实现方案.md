## 一、背景与目标

### 1.1 现状

当前 Xuflow 的工具系统是**封闭的**：[ToolRegistry](packages/core/src/tools/mod.rs) 中所有工具必须在 Rust 编译期注册。用户无法添加自定义工具，社区生态的工具也无法接入。

```rust
// 现状：所有工具硬编码注册
registry.register(Box::new(ReadFileTool));
registry.register(Box::new(WriteFileTool));
registry.register(Box::new(BashTool));
// ... 新增工具 = 写 Rust + 编译 + 发版
```

### 1.2 目标

实现 MCP client，让 Xuflow 能连接任意 MCP Server，将外部工具动态注册到 `ToolRegistry`，在 Agent 循环中透明调用。

### 1.3 MCP 简介

MCP 是 Anthropic 主导的开放标准协议，定义 LLM 应用与外部工具/数据源之间的通信规范：

| 组件 | 说明 |
|---|---|
| **MCP Server** | 独立进程（或远端服务），暴露 `tools`/`resources`/`prompts` |
| **MCP Client** | 嵌入在宿主应用（Xuflow）中，与 Server 通信 |
| **Transport** | 两种：`stdio`（本地子进程）和 `SSE + HTTP`（远端服务） |
| **协议层** | JSON-RPC 2.0，方法包括 `tools/list`、`tools/call`、`initialize` 等 |

## 二、实现后的效果

### 2.1 用户视角

**配置文件**（`%APPDATA%/xuflow/mcp_servers.json` 或项目根 `xuflow.json`）：

```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-postgres", "postgresql://localhost/mydb"]
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-filesystem", "/home/user/projects"]
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-brave-search"],
      "env": { "BRAVE_API_KEY": "xxx" }
    }
  }
}
```

**对话中的效果**：

```
用户: 查一下 orders 表最近一周的订单量，按地区分组

Agent 调用 MCP 工具:
  → mcp__postgres__execute_sql(sql="SELECT region, COUNT(*) FROM orders WHERE ...")
  → 返回查询结果

用户: 帮我重构 src/utils/ 下的所有 TypeScript 文件，把 any 改成具体类型

Agent 调用:
  → mcp__filesystem__list_directory(path="/home/user/projects/src/utils")
  → read_file / grep 分析类型
  → edit 逐个修复
```

### 2.2 开发者视角

| | 无 MCP | 有 MCP |
|---|---|---|
| 新增工具 | 写 Rust impl Tool，编译发版 | 用户装一个 npm 包，改一行配置 |
| 工具生态 | 自维护 16 个 | 社区数千个 MCP Server 直接可用 |
| 企业私有系统 | 无法接入 | 写个小脚本暴露为 MCP Server |
| Xuflow 定位 | 封闭工具集 | 开放 Agent 平台 |

### 2.3 现有工具与 MCP 工具的关系

MCP 工具与内置工具**共存**于同一个 `ToolRegistry`。命名规则：

- 内置工具：保持原名（`read_file`、`bash`、`web_search`...）
- MCP 工具：带 server 前缀避免冲突（`mcp__<server_name>__<tool_name>`）

Agent 循环不需要知道工具来源——它统一从 `ToolRegistry::list()` 获取所有工具定义，调用 `ToolRegistry::get()` 执行。

## 三、MCP 协议要点（与本实现相关的部分）

### 3.1 生命周期

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ 启动      │ ──→ │ 初始化    │ ──→ │ 运行中    │ ──→ │ 关闭      │
│ spawn 子   │     │ initialize │     │ tools/list │     │ 发送 EOF  │
│ 进程/连接  │     │ 握手       │     │ tools/call │     │ 或断开    │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
```

### 3.2 关键 JSON-RPC 方法（Client → Server）

| 方法 | 说明 |
|---|---|
| `initialize` | 握手，交换协议版本和能力声明 |
| `notifications/initialized` | Client 通知 Server 初始化完成 |
| `tools/list` | 获取 Server 提供的工具列表 |
| `tools/call` | 调用指定工具，传入 name + arguments |

### 3.3 Transport

#### stdio Transport

```
Xuflow                    MCP Server 进程
  │                            │
  │── stdin ──────────────────→│  (JSON-RPC request, 换行分隔)
  │←─ stdout ─────────────────│  (JSON-RPC response, 换行分隔)
  │                            │
```

推荐用于本地工具（filesystem、database、git 等）。Xuflow 作为父进程管理 Server 生命周期。

#### SSE + HTTP Transport

```
Xuflow                    MCP Server (远端)
  │                            │
  │── POST /message ──────────→│  (JSON-RPC request)
  │←─ SSE /events ─────────────│  (JSON-RPC response / notifications)
  │── GET /sse ───────────────→│  (建立 SSE 连接)
```

用于远端服务或需要持久连接的场景。Xuflow 已有 SSE 解析代码（`openai_compat_chat`），可复用。

### 3.4 工具描述格式

Server 返回的 `tools/list` 结果：

```json
{
  "tools": [
    {
      "name": "execute_sql",
      "description": "Execute a SQL query on the connected database",
      "inputSchema": {
        "type": "object",
        "properties": {
          "sql": { "type": "string", "description": "The SQL query to execute" }
        },
        "required": ["sql"]
      }
    }
  ]
}
```

这个格式与 OpenAI Function Calling 的 `parameters` 字段兼容，可以直接映射到 `FunctionDef.parameters`。

### 4.3 关键设计决策

#### 决策 1：每个 MCP Server 一个 `McpClient` 实例

`McpClient` 持有 transport 连接、已发现工具列表、生命周期状态。`McpManager` 管理多个 client。

#### 决策 2：`McpToolAdapter` 实现 `Tool` trait

```rust
/// 将 MCP 远程工具包装为本地 Tool trait。
/// 每个 MCP 工具一个 adapter 实例。
pub struct McpToolAdapter {
    server_name: String,    // 例如 "postgres"
    tool_name: String,      // 例如 "execute_sql"
    description: String,    // MCP Server 提供的描述
    parameters: serde_json::Value,  // JSON Schema
    is_dangerous: bool,     // 默认 true，可在配置中覆盖
    client: Arc<McpClient>, // 指向所属连接（用于 tools/call）
}
```

`execute()` 时，adapter 向 `McpClient` 发送 `tools/call` JSON-RPC 请求，等待结果返回。

#### 决策 3：MCP 工具默认标记为 dangerous

因为 MCP 工具的能力不可预知，首次调用必须经用户审批。后续可通过持久化权限系统（未来实现）白名单。

#### 决策 4：懒加载 vs 启动时加载

采用**启动时连接 + 发现**策略：
- 应用启动时一次性连接所有配置的 MCP Server
- `tools/list` 发现工具，批量注册到 `ToolRegistry`
- 连接失败不阻塞启动——在 UI 中显示警告

#### 决策 5：Transport 抽象

```rust
#[async_trait]
pub trait McpTransport: Send + Sync {
    /// 发送 JSON-RPC 请求，返回原始响应 JSON。
    async fn send(&self, request: serde_json::Value) -> Result<serde_json::Value, McpError>;
    /// 关闭连接，清理资源。
    async fn close(&self) -> Result<(), McpError>;
}
```

两种实现：
- `StdioTransport`：spawn 子进程，stdin 写 JSON，stdout 逐行读 JSON
- `SseTransport`：HTTP POST 发请求，SSE 连接收响应（复用现有 SSE 解析逻辑）

## 五、分阶段实施

### Phase 1：协议核心 + 单 Transport（2 天）

**范围**：MCP 协议基础 + stdio transport

#### 1.1 新增 `mcp/types.rs` — 协议类型

```rust
// JSON-RPC 2.0 基础类型
struct JsonRpcRequest {
    jsonrpc: String,        // "2.0"
    id: u64,
    method: String,
    params: Option<serde_json::Value>,
}

struct JsonRpcResponse {
    jsonrpc: String,
    id: u64,
    result: Option<serde_json::Value>,
    error: Option<JsonRpcError>,
}

// MCP 协议特定类型
struct InitializeResult {
    protocol_version: String,
    server_info: ServerInfo,
    capabilities: ServerCapabilities,
}

struct ToolDescription {
    name: String,
    description: String,
    input_schema: serde_json::Value,  // JSON Schema
}

struct ToolsListResult {
    tools: Vec<ToolDescription>,
}

struct ToolsCallParams {
    name: String,
    arguments: serde_json::Value,
}

struct ToolsCallResult {
    content: Vec<ToolContent>,  // text / image / embedded resource
    is_error: Option<bool>,
}
```

#### 1.2 新增 `mcp/transport/stdio.rs` — 本地子进程通信

```rust
pub struct StdioTransport {
    child: Mutex<Child>,                 // tokio::process::Child
    stdin: Mutex<ChildStdin>,
    stdout: Mutex<BufReader<ChildStdout>>,
    request_id: AtomicU64,
    pending_requests: Mutex<HashMap<u64, oneshot::Sender<serde_json::Value>>>,
}
```

核心流程：
1. `spawn` 子进程（`command` + `args` + `env`）
2. 后台 tokio task 持续读 stdout 逐行 JSON
3. 根据 `id` 匹配 `pending_requests`，通过 oneshot channel 返回
4. `send()` 写 stdin（写入 JSON + `\n`）

#### 1.3 新增 `mcp/client.rs` — 单 Server 连接管理

```rust
pub struct McpClient {
    name: String,                       // server 名称
    transport: Box<dyn McpTransport>,
    tools: Mutex<Vec<ToolDescription>>, // 已发现的工具
    capabilities: ServerCapabilities,   // Server 能力声明
}
```

方法：
- `async fn initialize(transport) -> Result<McpClient>` — 握手
- `async fn discover_tools() -> Vec<ToolDescription>` — tools/list
- `async fn call_tool(name, args) -> ToolsCallResult` — tools/call
- `async fn shutdown()` — 优雅关闭

#### 1.4 新增 `mcp/tool_adapter.rs` — 包装为 Tool trait

`McpToolAdapter` 结构体（见 4.3 决策 2），实现 `Tool` trait：

- `name()` → `"mcp__{server_name}__{tool_name}"`
- `description()` → 带 `[MCP: {server_name}]` 标记
- `parameters()` → `input_schema` 原样映射
- `is_dangerous()` → 默认 `true`
- `execute(args)` → 调 `client.call_tool(name, args)`，结果转为 `ToolResult`

#### 1.5 新增 `mcp/config.rs` — 配置反序列化

```rust
#[derive(Deserialize)]
pub struct McpServersConfig {
    pub mcp_servers: HashMap<String, McpServerConfig>,

#[derive(Deserialize)]
pub struct McpServerConfig {
    pub command: String,                    // 可执行文件
    pub args: Option<Vec<String>>,          // 命令行参数
    pub env: Option<HashMap<String, String>>, // 环境变量
    pub disabled: Option<bool>,             // 是否禁用
    pub dangerous_tools: Option<Vec<String>>, // 标记为非危险的工具名列表（可选豁免）

```

配置文件查找顺序（后者覆盖前者）：
1. `$APPDATA/xuflow/mcp_servers.json` — 全局
2. `{project_root}/xuflow.json` 中的 `mcpServers` 字段 — 项目级

#### 1.6 新增 `mcp/mod.rs` — McpManager

```rust
pub struct McpManager {

impl McpManager {
```

### Phase 3：SSE Transport + 健壮性（1.5 天）

#### 3.1 新增 `mcp/transport/sse.rs`

复用 `openai_compat_chat` 中的 SSE 解析逻辑：
- 启动时 `GET /sse` 建立事件流
- JSON-RPC 请求通过 `POST /message` 发送
- 响应通过 SSE 事件回传

#### 3.2 错误处理与重连

- Transport 断连时自动重试（指数退避，最多 3 次）
- 工具调用超时（默认 60 秒）返回错误给 LLM
- Server 崩溃时优雅降级——该 Server 的工具标记为不可用，不清除

#### 3.3 审批集成

MCP 工具首次调用时：
1. 触发 `ApprovalRequired` 事件（与现有危险工具审批相同流程）
2. 弹窗显示：工具名 + Server 名 + 参数
3. 用户选择：批准本次 / 批准本会话 / 拒绝
4. 本会话批准的 MCP 工具记录在内存 HashMap 中，后续自动通过

### Phase 4：CLI 接入（0.5 天）

CLI 通过 napi-rs 已经能使用 `xuflow-core`，只需：

1. napi 绑定暴露 `McpManager` 的构造函数和结果
2. CLI 启动时传入 `mcp_servers.json` 路径
3. CLI 的工具审批在终端用 y/n 交互

## 七、不在此次范围内的能力

以下能力属于后续迭代方向，本次不实现：

| 能力 | 原因 |
|---|---|
| **MCP Resources** (`resources/list`, `resources/read`) | 初版聚焦 tools；resources 用于注入上下文（如文档、schema），与 RAG 方案可互补 |
| **MCP Prompts** (`prompts/list`, `prompts/get`) | 预置 prompt 模板，优先级低于 tools |
| **持久化权限系统** | 另有专项方案；MCP 工具审批沿用现有 `ApprovalHandler` |
| **MCP Server 热重载** | 需重启应用生效；热重载在 hook 系统之后就绪 |
| **远程 MCP Server 认证** | SSE transport 先支持无认证/API Key 两种 |
| **Sampling** (`sampling/createMessage`) | Server 反向调 LLM，安全风险高，暂不开放 |

## 九、验收标准

1. 在 `mcp_servers.json` 中配置 `filesystem` MCP Server，启动 app
2. 发送消息 "列出桌面上的所有文件"，Agent 自动调用 `mcp__filesystem__list_directory`
3. 危险工具审批弹窗正常弹出，批准/拒绝行为正确
4. `mcp_servers.json` 中的 `disabled: true` 可禁用某个 Server
5. 配置了不存在的 command 时，不影响应用启动，设置页显示错误状态
6. 多个 MCP Server 同时连接，工具不冲突
