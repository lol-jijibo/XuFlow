# 会话自动摘要功能

> 新增日期：2026-06-17

## 概述

每次 AI 完成回复后，系统自动评估是否需要为当前会话生成/更新标题，将默认的"新会话 N"替换为概括对话内容的简短标题，显示在侧边栏项目列表中。

## 业务规则

### 触发时机

`agent:done` 事件触发后（即 AI 完成一轮回复），立即执行评估。

### 决策流程

```
agent:done 触发
    │
    ├─ 会话 titleSource === "manual" ？ → 跳过（用户手动命名，不覆盖）
    │
    ├─ 0 条用户消息 ？ → 跳过
    │
    ├─ 1 条用户消息
    │   ├─ 内容 < 100 字符 → 直接用提示词作标题（截断至 60 字符）
    │   └─ 内容 ≥ 100 字符 → 调用 LLM 生成摘要标题
    │
    └─ ≥ 2 条用户消息 → 调用 LLM 综合总结整个对话
```

### 标题来源标记 (`titleSource`)

| 值 | 含义 | 何时设置 |
|---|---|---|
| `"default"` | 系统默认标题（"新会话 N"） | 创建会话时未提供标题 |
| `"auto"` | AI 自动生成 | 摘要功能生成/短提示词直接引用 |
| `"manual"` | 用户手动输入 | 侧边栏新建会话时填写了自定义名称 |

**保护机制**：`source === "manual"` 的标题永远不会被自动摘要覆盖。

### 短提示词阈值

- **100 字符**：低于此值视为"短提示词"，直接用原文作标题
- **60 字符**：直接引用时的最大标题长度，超出则截断并加 `…`

这些阈值定义在 `useConversationSummary.ts` 的常量中，可按需调整。

## 架构设计

### 数据流

```
ChatPanel.vue
  │ 用户发送消息
  ▼
agent.ts → invoke("send_message")
  │
  ▼
Rust: AgentLoop.run()
  │ 流式返回 text-delta / tool-call / ...
  ▼
useTauriEvent.ts
  │ 监听 agent:done
  ├─ 标记消息完成
  └─ trySummarizeConversation()  ← 新增
       │
       ├─ 1条短消息 → 直接设标题
       └─ 需LLM → invoke("generate_title")
            │
            ▼
         Rust: generate_title command
            │ 构建摘要 prompt
            ▼
         LLM API (非流式，无工具)
            │ 返回标题文本
            ▼
         projectStore.updateConversationTitle()
```

### 为什么摘要走独立命令而非复用 AgentLoop？

| 维度 | AgentLoop.send_message | generate_title |
|---|---|---|
| 工具调用 | 注册 6 个工具 | 无工具 |
| 流式输出 | 逐字推送到前端 | 内部收集，只返回最终结果 |
| 审批 | 危险工具需用户审批 | 不需要 |
| 消息历史 | 追加到 AgentLoop 内部 history | 独立临时消息，不影响对话 |
| max_tokens | 4096 | 60（标题不需要长文本） |
| temperature | 0.7 | 0.3（更确定性的输出） |

### 并发控制

`trySummarizeConversation()` 内部使用模块级 `summarizing` 标志位，确保同一时间只有一个摘要请求在执行。

## 摘要提示词

### 系统提示词（System Prompt）

```
You are a title generator. Generate a concise, descriptive title
(max 30 characters) for the conversation. Return ONLY the title —
no quotes, no explanations, no prefixes.
```

### 单条长消息场景

```
Generate a short title (max 30 chars) for a conversation that
starts with this query:

{用户消息全文}

Title:
```

### 多轮对话场景

```
Generate a short title (max 30 chars) that captures the main
topic of this conversation:

User: {用户消息1（超过300字符则截断）}
Assistant: {AI回复1（超过300字符则截断）}
User: {用户消息2}
...

Title:
```

## 边界情况处理

| 场景 | 处理方式 |
|---|---|
| API Key 未配置 | `generate_title` 调用失败 → catch 捕获 → 静默失败，保留原标题 |
| 用户在中途切换会话 | `trySummarizeConversation` 开头已捕获 project/conv 引用；`updateConversationTitle` 会检查会话是否仍存在 |
| 用户手动改名 | `titleSource === "manual"` 检查阻止覆盖 |
| 旧数据无 titleSource | 字段为 optional (`?`)，旧数据加载正常；首次摘要会正常覆盖（titleSource 为 undefined ≠ "manual"） |
| LLM 返回引号/换行 | 前端清洗：去除首尾引号、中文引号、换行符，截断至 60 字符 |
| 快速连续发送消息 | `summarizing` 标志位防止并发重复请求 |
| Rust 侧消息超过 300 字符 | `build_summary_prompt` 自动截断，末尾追加 `...` |

## 未来可优化方向

1. **专用摘要模型**：使用更便宜的小模型（如 `deepseek-chat`）做摘要，不消耗主模型额度
2. **增量摘要**：多轮对话中，用上一轮的摘要 + 新消息生成更新标题，而非每次传全部历史
3. **摘要缓存**：同一对话内容不重复请求 LLM
4. **用户可配置阈值**：在设置页面暴露短提示词长度阈值和标题最大长度
