# Tier 1 工具实现计划

## 背景

Xuflow 当前有 6 个基础工具（read_file / write_file / list_dir / grep / bash / web_fetch），缺少精确编辑、文件查找、任务追踪、Git 集成和规划能力。要实现"调用 API 完成完整项目编写"，需要补齐这 5 项能力。

## 架构模式（已有，直接复用）

- **Rust 工具**: 在 `packages/core/src/tools/` 下新增文件 → 实现 `Tool` trait → 在 `mod.rs` 导出 → 在 `desktop/src-tauri/src/commands/chat.rs` 的 `build_agent()` 中注册
- **危险工具审批**: `is_dangerous() = true` → Agent 循环自动发送 `ApprovalRequired` 事件 → 前端 `ApprovalModal.vue` 弹出 → 用户点击后通过 `respond_approval` command 返回结果
- **前端事件桥接**: Rust 侧 `app.emit("agent:xxx", payload)` → `useTauriEvent.ts` 中 `listen("agent:xxx", ...)` 处理
- **依赖**: `walkdir` 已可用（glob 匹配可复用）；`regex` 已可用；`std::process::Command` 模式已在 `bash.rs` 中验证

## 一、Edit 工具（精确字符串替换）

### 需要修改/新增的文件
1. **新建** `packages/core/src/tools/edit.rs` — 核心实现
2. **编辑** `packages/core/src/tools/mod.rs` — 添加 `pub mod edit;`
3. **编辑** `desktop/src-tauri/src/commands/chat.rs` — 注册工具
4. **编辑** `desktop/src/components/approval/ApprovalModal.vue` — 增强审批界面（显示 diff 预览）
5. **编辑** `packages/core/src/agent/system_prompt.rs` — 告知模型如何使用

### 实现细节
- **参数**: `path` (string), `old_string` (string), `new_string` (string), `replace_all` (bool, optional, default false)
- **逻辑**:
  1. 读取文件内容
  2. 查找 `old_string` 出现次数
  3. 0 次 → 返回 error（防止幻觉写错位置）
  4. >1 次且 `replace_all=false` → 返回 error + 所有出现的行号
  5. 1 次或 `replace_all=true` → 替换 → write 回文件
  6. 确保 parent dir 存在（复用 `WriteFileTool` 的 create_dir_all 模式）
- **危险级别**: `true`（修改用户文件）
- **审批增强**: 在 ApprovalModal 中，当 tool 为 `edit` 时，将 old_string 渲染为红色删除线、new_string 渲染为绿色添加 —— 给用户直观的 diff 感受

### 复杂度评估: 中（核心逻辑简单，前端 diff 展示需要一点 CSS 工作）

## 二、Glob 工具（文件模式匹配）

### 需要修改/新增的文件
1. **新建** `packages/core/src/tools/glob.rs` — 核心实现
2. **编辑** `packages/core/src/tools/mod.rs` — 添加 `pub mod glob;`
3. **编辑** `desktop/src-tauri/src/commands/chat.rs` — 注册工具
4. **编辑** `packages/core/Cargo.toml` — 添加 `glob = "0.3"` 依赖
5. **编辑** `packages/core/src/agent/system_prompt.rs` — 告知模型

### 实现细节
- **参数**: `pattern` (string, 如 `**/*.rs`), `path` (string, optional, 默认 `.`)
- **逻辑**:
  1. 使用 `glob::glob()` 匹配文件
  2. 过滤掉隐藏文件和 `node_modules`/`target`/`.git`（复用 grep.rs 的过滤模式）
  3. 返回排序后的文件路径列表
  4. 限制最大结果数（如 1000）
- **危险级别**: `false`（只读）
- **替代方案**: 如果不想引入 glob crate，可以用 `walkdir` + 手动 glob 匹配，但 `glob` crate 更健壮且零成本（纯 Rust 库）

### 复杂度评估: 低（模式清晰，逻辑简单）

## 三、TodoWrite 工具（任务追踪）

### 需要修改/新增的文件
1. **新建** `packages/core/src/tools/todo.rs` — 工具实现
2. **编辑** `packages/core/src/tools/mod.rs` — 添加 `pub mod todo;`
3. **编辑** `packages/core/src/backends/mod.rs` — 新增 `StreamEvent::TodoUpdate` 变体
4. **编辑** `packages/core/src/agent/loop_.rs` — 在工具执行后检测 todo_write 并发射 TodoUpdate 事件
5. **编辑** `desktop/src-tauri/src/commands/chat.rs` — 注册工具 + 转发 TodoUpdate 事件到前端
6. **编辑** `desktop/src/composables/useTauriEvent.ts` — 新增 `agent:todo-update` 监听
7. **编辑** `desktop/src/stores/agent.ts` — 新增 `todos` 状态
8. **新建** `desktop/src/components/chat/TodoPanel.vue` — 任务列表面板
9. **编辑** `desktop/src/components/chat/ChatPanel.vue` — 引入 TodoPanel，放在聊天区旁边或折叠面板中

### 实现细节
- **参数**: `todos` (array of `{content: string, status: "pending"|"in_progress"|"completed"}`)
- **逻辑**: 不修改任何文件，只返回成功确认。真正的作用是通过 Agent 循环检测到此工具调用后，发射 `TodoUpdate` 事件到前端
- **Agent 循环改动**:
  ```rust
  // loop_.rs 中，execute 之后：
  if tool_name == "todo_write" {
      if let Ok(todos) = serde_json::from_str(&result.content) {
          tx.send(StreamEvent::TodoUpdate { todos }).await.ok();
      }
  }
  ```
- **危险级别**: `false`（不触碰用户文件）
- **前端 UI**: 
  - TodoPanel 作为 ChatPanel 的一个可折叠侧边面板（或放在 chat-body 顶部的横条）
  - 显示当前所有任务的 checkbox 列表
  - completed 的任务显示删除线
  - in_progress 的任务高亮

### StreamEvent 新变体:
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TodoItem {
    pub content: String,
    pub status: String, // "pending" | "in_progress" | "completed"
}

// 在 StreamEvent enum 中添加：
TodoUpdate { todos: Vec<TodoItem> },
```

### 复杂度评估: 高（涉及 agent loop、StreamEvent、前端新组件三层改动）

## 四、Git 工具集

### 需要修改/新增的文件
1. **新建** `packages/core/src/tools/git.rs` — 5 个 git 工具
2. **编辑** `packages/core/src/tools/mod.rs` — 添加 `pub mod git;`
3. **编辑** `desktop/src-tauri/src/commands/chat.rs` — 注册 5 个工具
4. **编辑** `packages/core/src/agent/system_prompt.rs` — 告知模型

### 工具清单

| 工具 | 参数 | 危险 | 说明 |
|------|------|------|------|
| `git_status` | `working_dir` | 否 | 执行 `git status --porcelain` |
| `git_diff` | `working_dir`, `staged`(optional) | 否 | `git diff` 或 `git diff --staged` |
| `git_log` | `working_dir`, `count`(optional, default 10) | 否 | `git log --oneline -n <count>` |
| `git_add` | `working_dir`, `files` (string[]) | 是 | `git add <files>` |
| `git_commit` | `working_dir`, `message` (string) | 是 | `git commit -m "<message>"` |

### 实现细节
- **执行方式**: 复用 `bash.rs` 的 `std::process::Command` 模式，直接调用系统 git
- **路径安全**: 
  - 验证 `working_dir` 是有效目录
  - 对于 `git_add`，验证文件参数中的文件存在
- **提交消息审批**: `git_commit` 为危险操作，触发审批弹窗，用户可以在弹窗中看到并修改 commit message
- **错误处理**: 如果 git 未安装或不在 git 仓库中，返回清晰的错误信息

### 复杂度评估: 中（5 个工具但每个都很薄，只是对 git CLI 的封装）

## 五、Plan Mode（规划模式）

### 需要修改/新增的文件
1. **新建** `packages/core/src/tools/plan.rs` — `propose_plan` 工具
2. **编辑** `packages/core/src/tools/mod.rs` — 添加 `pub mod plan;`
3. **编辑** `packages/core/src/agent/system_prompt.rs` — 添加规划模式的指令
4. **编辑** `packages/core/src/backends/mod.rs` — 新增 `StreamEvent::PlanProposed` 变体
5. **编辑** `desktop/src-tauri/src/commands/chat.rs` — 注册工具 + 转发事件
6. **编辑** `desktop/src/composables/useTauriEvent.ts` — 监听 `agent:plan-proposed`
7. **编辑** `desktop/src/stores/agent.ts` — 新增 `pendingPlan` 状态
8. **新建** `desktop/src/components/approval/PlanApprovalCard.vue` — 计划审批卡片
9. **编辑** `desktop/src/components/chat/ChatPanel.vue` — 添加 Plan Mode 开关 + 集成 PlanApprovalCard

### 实现细节

**propose_plan 工具**:
- **参数**: `title` (string), `steps` (string[]), `files_to_modify` (string[])
- **危险级别**: `true`（需要用户审批才能继续）
- **逻辑**: 将 plan 存储到 tool result content 中（JSON），Agent 循环检测到此工具后发射 `PlanProposed` 事件

**StreamEvent 新变体**:
```rust
PlanProposed {
    title: String,
    steps: Vec<String>,
    files_to_modify: Vec<String>,
},
```

**Agent 循环改动**（在 loop_.rs 中）:
- 当检测到 `propose_plan` 工具调用时，暂停执行，发送 `PlanProposed` 事件
- 等待用户审批（通过 `respond_approval` 机制复用，因为 `is_dangerous() = true`）
- 如果批准 → 继续执行后续工具调用
- 如果拒绝 → 返回 error 告诉模型修改计划

**前端 UI**:
- ChatPanel footbar 添加 "Plan Mode" 开关（与 "自动执行" 并排）
- 当 `pendingPlan` 不为 null 时，在消息流中插入 PlanApprovalCard
- PlanApprovalCard 显示：标题、步骤列表（编号）、涉及文件、批准/拒绝按钮
- 审批通过后，卡片收起为 "✅ 计划已批准" 的摘要状态

**系统提示词更新**: 当 Plan Mode 开启时，追加：
```
Before writing any code, first explore the codebase to understand existing patterns,
then call propose_plan with a detailed implementation plan. Wait for user approval
before executing any changes.
```

### 复杂度评估: 中-高（涉及 agent loop 改动 + 新事件类型 + 新前端组件）

## 实施顺序建议

```
第 1 步: Glob   (最简单，验证工具添加流程 → 建立信心)
第 2 步: Edit   (最核心，直接影响代码编写能力)
第 3 步: Git    (3 个安全工具 + 2 个危险工具，批量封装)
第 4 步: TodoWrite (需要改动 agent loop + StreamEvent + 前端，但独立性强)
第 5 步: Plan Mode  (最复杂，依赖审批流程，需要 agent loop 改动)
```

## 验证方式

1. **每个工具完成后**: 在桌面端启动应用 → 向 Agent 发送触发该工具的指令 → 观察工具调用是否正确执行
2. **Edit 工具**: 发送 "把 src/xxx.rs 中的 foo 改成 bar" → 确认文件被正确修改
3. **Glob 工具**: 发送 "列出所有 .rs 文件" → 确认返回完整文件列表
4. **Git 工具**: 在 git 仓库中发送 "git status" → 确认返回状态
5. **TodoWrite**: 发送 "帮我实现 3 个功能" → 确认前端显示任务列表
6. **Plan Mode**: 开启 Plan Mode → 发送复杂需求 → 确认显示计划卡片 → 审批后执行
